# Kerwin Signal-Chain Research Skill

## 1. 目标

将一个可验证的细节扩展为一项具有深度、广度和持续跟踪价值的产业研究。

核心链路：

```text
细节发现 → 实体核验 → 证据分级 → 技术拆解 → 产业链扩展
→ 投资映射 → 领先指标 → 证伪条件 → 专题发布 → 持续监测
```

## 2. 触发条件

出现以下任一情况时使用本Skill：

- 电话会或财报材料中出现陌生客户、合作方、产品或技术名词；
- 用户要求“顺藤摸瓜”“按图索骥”“继续追踪这个细节”；
- 单一客户案例可能映射至更广泛的基础设施或产业链需求；
- 新芯片、机柜、网络、电力或存储方案可能造成系统性升级；
- 需要把零散研究讨论整理为结构化网页、报告或监测清单；
- 需要严格区分事实、自报数据、推断和传闻。

## 3. 不适用场景

- 只需要快速查询一个稳定事实；
- 只需要复述财报数字而不要求产业链扩展；
- 缺乏任何可核验原始线索；
- 涉及未授权客户信息、内部邮件、账户数据或其他非公开材料；
- 用户要求将未经核验的传闻直接写成事实。

## 4. 研究原则

### 4.1 从细节出发，不从结论出发

先确认管理层、文件或客户案例具体说了什么，再讨论产业意义。不得先写宏大结论，再选择性寻找支持材料。

### 4.2 顺藤摸瓜，按图索骥

```text
一个名字或产品
  ↓
它是谁／是什么
  ↓
为什么在此时被提到
  ↓
对应什么工作负载和真实需求
  ↓
需要哪些芯片、机柜、网络、存储和电力
  ↓
订单可能传导到哪些企业
  ↓
未来用什么信号验证或证伪
```

### 4.3 先证据，后结论

每项关键陈述必须标注证据等级。

| 等级 | 类型 | 使用方式 |
|---|---|---|
| A | 财报、电话会原文、监管文件、官网、官方产品文档 | 可以直接陈述并保留原始口径 |
| B | 企业或合作方自报性能、效率、成本和客户结果 | 必须注明测试条件或自报属性 |
| C | 研究推断 | 使用“我们判断”“可能”“尚待验证” |
| D | 网传、二手整理、匿名消息 | 明确非官方、转述误差和不可逐字归因 |

### 4.4 严格区分成熟度阶段

以下阶段不得混写：

```text
概念／路线图
→ 芯片样片
→ 工程样机
→ 完整系统Bring-up
→ 客户Early Access
→ Preview
→ GA
→ 规模交付
→ 利用率与收入确认
```

### 4.5 区分技术与财务受益

分别判断：

- 技术受益；
- 订单受益；
- 收入受益；
- 毛利和利润受益；
- 估值受益；
- 是否已经被市场定价。

## 5. 标准任务流

### Step 1：提取原始线索

记录：

- 原始表述；
- 出处和时间；
- 发言人或文件；
- 上下文；
- 初步研究价值；
- 证据等级。

### Step 2：实体核验

#### 公司

- 成立时间、总部、创始团队；
- 融资、股东和董事；
- 产品或研究方向；
- 商业化阶段；
- 与原始披露主体的关系。

#### 人物

- 准确职位和履历；
- 代表项目；
- 是否为创始人、技术负责人或顾问；
- 避免把“核心技术领导者”夸大成“创始人”。

#### 产品

- 自研还是第三方；
- 芯片、机架、云实例还是完整服务；
- 当前处于计划、测试、Preview、GA或规模交付哪一步。

### Step 3：建立事实关系图

```text
研究主体
├── 创始团队
├── 技术路线
├── 资金来源
├── 客户与合作方
├── 云平台
├── 加速器与CPU
├── 网络与互联
├── 内存与存储
├── 电力与散热
└── 商业化与验证节点
```

### Step 4：拆解工作负载

至少回答：

1. AI系统具体在做什么？
2. 训练、推理、Agent、仿真和评估如何连接？
3. CPU、GPU、网络、内存和存储分别承担什么任务？
4. 主要瓶颈是算力、带宽、容量、功率、数据还是软件？
5. 新系统改善的是绝对性能、单位电力效率、可扩展性还是交付能力？

### Step 5：扩展产业链

按照以下顺序检查：

- **计算：** GPU、TPU、CPU、HBM、DDR、先进封装；
- **互联：** NVLink、交换芯片、以太网／InfiniBand、1.6T、硅光、CPO；
- **存储：** 企业SSD、NAND、并行文件系统、Checkpoint、KV Cache、经验回放；
- **电力与散热：** 变压器、整流、母线、800V HVDC、液冷、CDU；
- **软件：** 训练与推理框架、调度、模拟器、验证器、Agent编排和数据治理。

每一项必须标记为：

- 核心标准配置；
- 可选配置；
- 相邻升级方向；
- 尚未确认的潜在需求。

### Step 6：建立投资映射

对每个节点回答：

| 维度 | 问题 |
|---|---|
| 需求来源 | 训练、推理、Agent、持续学习还是Physical AI？ |
| 增量形态 | 数量、带宽、功率、容量、使用时长还是软件收入？ |
| 受益主体 | 芯片、设备、零部件、云厂商、软件或服务？ |
| 财务传导 | 订单何时进入收入、毛利和现金流？ |
| 市场定价 | 已充分定价、部分定价还是尚未关注？ |

### Step 7：建立产业信号灯

#### 绿灯

- 芯片和完整机架按期交付；
- 客户进入生产环境；
- 单位电力或单位成本性能改善；
- 利用率、订单、收入和现金流开始兑现。

#### 黄灯

- 只有样机或Benchmark；
- 仅少数Early Access客户；
- 软件、网络、电力或散热尚未完成规模验证；
- 性能只在特定模型和配置中成立。

#### 红灯

- 交付延期；
- 客户削减资本开支；
- 大规模集群扩展失败；
- 性能无法在生产环境复制；
- 库存、利用率或单位经济性恶化。

### Step 8：形成标准研究输出

必须包含：

1. 研究起点；
2. 为什么这个细节重要；
3. 实体画像；
4. 技术与工作负载；
5. 事实、推断和争议的分层；
6. 产业链扩展；
7. 投资映射；
8. 已确认与尚未确认；
9. 领先指标和证伪条件；
10. 后续监测清单；
11. 原始来源。

### Step 9：网页或报告发布

推荐页面结构：

```text
标题与核心判断
→ 研究起点与方法论
→ 公司／技术主体
→ 工作原理与流程图
→ 路线对照
→ 基础设施架构
→ 产业信号灯
→ 投资映射
→ 后续验证点
→ 来源与口径
→ 结论
```

发布前检查：

- 移动端可读；
- 深色背景使用浅色文字；
- 英文术语附中文解释；
- 不把相邻方向写成标准配置；
- 不把计划写成已完成；
- 不把系统级性能写成单芯片性能；
- 事实、推断和传闻有明确视觉区分。

### Step 10：持续监测

专题发布后，建立：

- 事件时间线；
- 当前状态灯；
- 下一次验证日期或事件；
- 正向催化剂；
- 风险和证伪条件；
- 需要更新的网页段落和首页摘要。

## 6. 输出质量标准

完成后，读者必须能够回答：

1. 线索最初从哪里发现？
2. 主体到底是谁或是什么？
3. 为什么现在重要？
4. 它改变了什么技术架构？
5. 需求如何沿产业链传导？
6. 哪些企业可能获得收入和利润？
7. 哪些事实已经确认？
8. 哪些仍是推断或预期？
9. 接下来跟踪什么？
10. 什么情况会证明原判断错误？

不能回答这十个问题，任务尚未完成。

## 7. 核心执行提示词

```text
使用Kerwin Signal-Chain Research方法分析以下线索。

先定位原始出处，并把关键陈述分为：
A. 官方直接事实；B. 企业自报结果；C. 研究推断；D. 网传或二手材料。

随后完成公司、人物和产品实体核验，严格区分概念、样片、样机、Early Access、Preview、GA、规模交付与收入确认。

按照“应用需求—工作负载—模型与算法—CPU/GPU—机柜—网络与光互联—内存与存储—电力与液冷—云平台和软件”的顺序拆解。

再把技术链转换为投资映射，分别判断技术受益、订单受益、收入受益、利润受益与估值受益。

最后输出：研究起点、因果链、已确认、尚未确认、产业信号灯、领先指标、证伪条件、后续监测清单和原始来源。

研究必须体现“顺藤摸瓜、按图索骥”：从一个细节扩展为完整产业链，从一次事件提炼出可持续验证的投资框架。
```

## 8. 隐私要求

- 仅使用用户明确授权的资料和公开信息；
- 不把客户身份、账户、持仓、交易、内部邮件或访问密钥写入公开输出；
- 发布前执行敏感信息检查；
- 公开Skill仓库与私人研究和网站仓库保持隔离。
